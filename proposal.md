# 分布式缓存系统 - Proposal

## 1. 背景

### 1.1 为什么需要这个服务

在现代分布式系统中，缓存已成为提升系统性能的关键技术。通过将热点数据缓存在内存中，可以显著减少数据库访问压力。然而，单机缓存存在容量限制、并发瓶颈等局限性。

本项目的目标是一个**轻量级的分布式缓存系统**，聚焦于：
1. **学习分布式缓存的核心原理**：LRU淘汰算法、一致性哈希、主从复制
2. **实践TCP网络编程**：自定义协议设计、多客户端并发、连接管理
3. **实现数据一致性**：分片机制、复制策略

### 1.2 学习目标

- 掌握LRU缓存淘汰算法的实现原理和优化技巧
- 理解一致性哈希在分布式系统中的应用
- 学习TCP协议设计和网络编程最佳实践
- 实践简化版的主从复制机制

## 2. 目标

### 2.1 总体目标

构建一个**轻量级、高性能、易扩展**的分布式内存缓存系统，核心目标是：

1. 实现基于LRU算法的缓存淘汰机制
2. 提供基于TCP协议的网络服务
3. 设计简洁高效的自定义二进制协议
4. 通过一致性哈希实现数据分片，支持水平扩展
5. 实现主从复制，提供数据冗余和高可用

### 2.2 具体功能目标

#### 2.2.1 LRU缓存淘汰策略
- 实现基于双向链表和哈希表的标准LRU算法
- 支持GET、SET、DELETE等基本缓存操作
- 精确控制缓存容量，当缓存满时淘汰最少使用的条目

#### 2.2.2 TCP服务器实现
- 构建高性能的TCP服务器，支持多客户端并发连接
- 实现goroutine-per-connection模型
- 处理网络异常和连接管理

#### 2.2.3 自定义协议设计
- 设计简洁的二进制协议，减少协议开销
- 支持多种命令类型（GET, SET, DELETE, INFO等）
- 实现请求/响应序列化和反序列化
- 定义清晰的协议格式和错误处理机制

#### 2.2.4 一致性哈希分片
- 实现虚拟节点机制，平衡各分片负载
- 支持动态添加/移除缓存节点，无需大规模数据迁移
- 使用FNV-1a哈希算法保证分布均匀性

#### 2.2.5 主从复制（简化版）
- 实现主从节点同步机制
- 支持主节点写操作同步到从节点
- 简单的复制策略（全量复制 + 写操作同步）

## 3. 范围

### 3.1 In Scope（包含的功能）

#### 核心功能（必须实现）
1. **LRU缓存淘汰算法**
   - 使用双向链表和哈希表实现标准LRU算法
   - 支持GET、SET、DELETE基本缓存操作
   - 实现容量限制和自动淘汰

2. **TCP服务器**
   - 支持多客户端并发连接
   - 实现简单的连接管理（接收、处理、关闭）
   - 处理网络异常（连接断开、超时）

3. **自定义二进制协议**
   - 设计简洁的二进制协议格式
   - 支持GET、SET、DELETE、INFO命令
   - 实现请求/响应序列化/反序列化
   - 定义错误码（SUCCESS, ERROR_UNKNOWN_COMMAND, ERROR_INVALID_KEY, ERROR_INVALID_VALUE, ERROR_CACHE_FULL）

4. **一致性哈希分片**
   - 实现一致性哈希环
   - 实现虚拟节点机制（每个物理节点默认100个虚拟节点）
   - 实现基本的分片数据路由

5. **简化版主从复制**
   - 主节点写操作同步到从节点
   - 从节点断开重连后请求全量同步
   - 主节点故障后从节点切换为新的主节点

### 3.2 Out of Scope（不包含的功能）

#### 已明确排除的功能
1. **高级网络特性**
   - 连接池管理
   - 心跳检测
   - 网络心跳机制
   - 异步I/O模型（协程/多线程）

2. **高级缓存功能**
   - 缓存统计和监控（命中率、读写次数等）
   - 批量操作
   - 过期时间（TTL）

3. **高级一致性哈希**
   - 节点动态增删（热插拔）
   - 分片数据迁移
   - 负载均衡优化
   - 节点自动发现

4. **高级复制特性**
   - 多从节点复制
   - 异步复制
   - 复制过滤
   - 增量复制

5. **持久化机制**
   - RDB快照
   - AOF日志
   - 数据持久化到磁盘

6. **高可用方案**
   - Raft/Paxos一致性算法
   - 自动故障检测和选举
   - 多主复制
   - 哨兵模式

7. **安全特性**
   - TLS/SSL加密
   - 认证和授权
   - 防止DDoS攻击
   - 输入验证和过滤

8. **高级数据结构**
   - Hash、List、Set、ZSet等复合数据结构
   - 位图、HyperLogLog等高级数据结构
   - 流式数据结构

9. **性能优化**
   - 内存池
   - 批量写入
   - Pipeline支持
   - 脚本执行

10. **运维工具**
    - 监控指标导出
    - 可视化面板
    - 命令行管理工具
    - 配置热加载

### 3.3 技术约束
- 单节点支持最多10万条缓存数据
- 支持1000+并发连接
- 响应延迟目标：< 1ms（缓存命中）
- 内存使用限制：单进程最多使用2GB内存

## 4. 验收标准

### 4.1 核心功能验收

#### 验收标准 1: LRU缓存淘汰算法正确性
**系统 SHALL** 正确实现标准LRU算法，验证条件：
- [x] 使用双向链表+哈希表实现LRU算法
- [x] 当缓存容量为100条时，添加第101条数据后，Key1被正确淘汰
- [x] 重复访问的Key保持在高频使用位置
- [x] DELETE操作后，被删除Key的位置不被视为最近使用
- [x] 查询不存在的Key返回null

#### 验收标准 2: TCP服务器基本功能
**系统 SHALL** 提供支持多客户端连接的TCP服务器，验证条件：
- [x] 服务器成功启动并监听端口7000
- [x] 支持至少10个客户端并发连接
- [x] 客户端连接后能成功发送请求并接收响应
- [x] 客户端强制断开后，服务器正确清理连接资源

#### 验收标准 3: 自定义二进制协议
**系统 SHALL** 正确实现自定义二进制协议，验证条件：
- [x] 协议格式清晰定义（Command+KeyLen+ValueLen+Data）
- [x] GET、SET、DELETE、INFO命令能正确处理
- [x] 对非法命令返回ERROR_UNKNOWN_COMMAND
- [x] 对缺少参数的请求返回ERROR_INVALID_KEY
- [x] 请求/响应序列化和反序列化正确

#### 验收标准 4: 一致性哈希分片
**系统 SHALL** 实现一致性哈希分片机制，验证条件：
- [x] 初始化时创建一致性哈希环
- [x] 每个物理节点有100个虚拟节点
- [x] 1000次SET操作后，3个分片的数据分布差异<30%
- [x] 同一个Key总是路由到同一个分片

#### 验收标准 5: 主从复制基本功能
**系统 SHALL** 实现简化版主从复制，验证条件：
- [x] Master的SET操作同步到Slave
- [x] Slave断开重连后，能从Master请求同步所有数据
- [x] Master故障后，Slave切换为新的Master
- [x] 原Master重启后，连接到新Master并同步数据

### 5.4 协议设计（简要）

#### 帧格式
```
+-------------------+----------------+---------------+------------------+
|   Command (1 byte) |   KeyLen (4B)   |   ValueLen (4B)  |    Data (Var)    |
+-------------------+----------------+---------------+------------------+
```

#### 命令码
- `0x01`: GET
- `0x02`: SET
- `0x03`: DELETE
- `0x04`: INFO

#### 错误码
- `0x00`: SUCCESS
- `0x01`: ERROR_UNKNOWN_COMMAND
- `0x02`: ERROR_INVALID_KEY
- `0x03`: ERROR_INVALID_VALUE
- `0x04`: ERROR_CACHE_FULL

## 6. 风险评估

### 6.1 技术风险

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| 协议设计不合理导致性能瓶颈 | 中 | 高 | 先设计协议草案，进行小规模性能测试验证 |
| 一致性哈希实现不当导致负载不均 | 中 | 中 | 使用成熟的哈希算法，编写测试用例验证负载分布 |
| 主从复制同步延迟高 | 低 | 中 | 限制数据大小，使用批量同步机制 |
| 内存泄漏风险 | 低 | 高 | 使用pprof监控内存，编写内存泄漏测试 |

### 6.2 实现风险

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| 功能范围蔓延 | 中 | 高 | 严格遵循范围定义，功能变更需重新评估 |
| 性能未达到预期 | 中 | 中 | 采用基准测试验证，必要时调整实现策略 |
| 测试覆盖率不足 | 低 | 高 | 编写单元测试和集成测试，确保覆盖率>80% |
