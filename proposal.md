# 分布式缓存系统 - Proposal

## 1. 背景

### 1.1 为什么需要这个服务

在现代分布式系统中，缓存已成为提升系统性能的关键技术。通过将热点数据缓存在内存中，可以显著减少数据库访问压力。然而，单机缓存存在容量限制、并发瓶颈等局限性。

本项目的目标是一个**轻量级的分布式缓存系统**，聚焦于：
1. **学习分布式缓存的核心原理**：LRU淘汰算法、一致性哈希、主从复制
2. **实践TCP网络编程**：自定义协议设计、多客户端并发、连接管理
3. **实现数据一致性**：分片机制、复制策略

### 1.2 学习目标

- 掌握LRU缓存淘汰算法的实现原理和优化技巧
- 理解一致性哈希在分布式系统中的应用
- 学习TCP协议设计和网络编程最佳实践
- 实践简化版的主从复制机制

## 2. 目标

### 2.1 总体目标

构建一个**轻量级、高性能、易扩展**的分布式内存缓存系统，核心目标是：

1. 实现基于LRU算法的缓存淘汰机制
2. 提供基于TCP协议的网络服务
3. 设计简洁高效的自定义二进制协议
4. 通过一致性哈希实现数据分片，支持水平扩展
5. 实现主从复制，提供数据冗余和高可用

### 2.2 具体功能目标

#### 2.2.1 LRU缓存淘汰策略
- 实现基于双向链表和哈希表的标准LRU算法
- 支持GET、SET、DELETE等基本缓存操作
- 精确控制缓存容量，当缓存满时淘汰最少使用的条目

#### 2.2.2 TCP服务器实现
- 构建高性能的TCP服务器，支持多客户端连接
- 实现连接池管理，避免频繁创建/销毁连接的开销
- 实现心跳机制，检测和维护活跃连接

#### 2.2.3 自定义协议设计
- 设计简洁的二进制协议，减少协议开销
- 支持多种命令类型（GET, SET, DELETE, INFO等）
- 实现请求/响应序列化和反序列化
- 定义清晰的协议格式和错误处理机制

#### 2.2.4 一致性哈希分片
- 实现虚拟节点机制，平衡各分片负载
- 支持动态添加/移除缓存节点，无需大规模数据迁移
- 实现分片级别的主节点选举

#### 2.2.5 主从复制（简化版）
- 实现主从节点同步机制
- 支持主节点故障时从节点提升为新主节点
- 简单的复制策略（全量复制+增量复制）

## 4. 验收标准

### 4.1 核心功能验收

#### 验收标准 1: LRU缓存淘汰算法正确性
**系统 SHALL** 正确实现标准LRU算法，验证条件：
- [ ] 使用双向链表+哈希表实现LRU算法
- [ ] 当缓存容量为100条时，添加第101条数据后，Key1被正确淘汰
- [ ] 重复访问的Key保持在高频使用位置
- [ ] DELETE操作后，被删除Key的位置不被视为最近使用
- [ ] 查询不存在的Key返回null

#### 验收标准 2: TCP服务器基本功能
**系统 SHALL** 提供支持多客户端连接的TCP服务器，验证条件：
- [ ] 服务器成功启动并监听端口7000
- [ ] 支持至少10个客户端并发连接
- [ ] 客户端连接后能成功发送请求并接收响应
- [ ] 客户端强制断开后，服务器正确清理连接资源

#### 验收标准 3: 自定义二进制协议
**系统 SHALL** 正确实现自定义二进制协议，验证条件：
- [ ] 协议格式清晰定义（帧头+长度+命令+数据）
- [ ] GET、SET、DELETE、INFO命令能正确处理
- [ ] 对非法命令返回ERROR_UNKNOWN_COMMAND
- [ ] 对缺少参数的请求返回ERROR_INVALID_KEY
- [ ] 请求/响应序列化和反序列化正确

#### 验收标准 4: 一致性哈希分片
**系统 SHALL** 实现一致性哈希分片机制，验证条件：
- [ ] 初始化时创建一致性哈希环
- [ ] 每个物理节点有100个虚拟节点
- [ ] 1000次SET操作后，3个分片的数据分布差异<30%
- [ ] 同一个Key总是路由到同一个分片
- [ ] 添加新节点后，只有约10-20%的数据迁移

#### 验收标准 5: 主从复制基本功能
**系统 SHALL** 实现简化版主从复制，验证条件：
- [ ] Master的SET操作在10ms内同步到Slave
- [ ] Slave断开重连后，能从Master请求同步所有数据
- [ ] Master故障后，Slave切换为新的Master
- [ ] 原Master重启后，连接到新Master并同步数据

### 5.4 协议设计（简要）

#### 帧格式
```
+-------------------+----------------+---------------+------------------+
|   Command (1 byte) |   KeyLen (4B)   |    ValueLen   |    Data (Var)    |
+-------------------+----------------+---------------+------------------+
```

#### 命令码
- `0x01`: GET
- `0x02`: SET
- `0x03`: DELETE
- `0x04`: INFO

#### 错误码
- `0x00`: SUCCESS
- `0x01`: ERROR_UNKNOWN_COMMAND
- `0x02`: ERROR_INVALID_KEY
- `0x03`: ERROR_INVALID_VALUE
- `0x04`: ERROR_CACHE_FULL
